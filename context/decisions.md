# Decision Log (ADR-style)

Each entry: the decision, why, and rejected alternatives. Append new decisions; don't rewrite. Locked decisions shouldn't be relitigated without a stated reason.

_All decisions below: 2026-06-22, from the end-to-end vision interview._

### D1 ‚ÄĒ Pure observer, not a coach
The app never interrupts (except an opt-in evening recap ping). **Why:** a calm mirror is maximally trustworthy and a distinct identity from the sibling "Nudge" app. **Rejected:** opt-in nudges (reintroduces productivity-guilt we cut), live menubar HUD (rejected as noise).

### D2 ‚ÄĒ Two-axis model: context √ó project
**Why:** "you gave usage_os a 90-min block" needs the project axis; context alone is too coarse. **Rejected:** contexts-only (vague recaps), projects-only (no productive/distracting read).

### D3 ‚ÄĒ The dial is the soul; ship order dial ‚Üí categorization ‚Üí recap
**Why:** the dial is the signature and works with zero AI ‚ÄĒ lowest-risk path to a real product. **Rejected:** recap-first (depends on the riskiest piece), treating them as inseparable (bigger, riskier v1).

### D4 ‚ÄĒ Capture: Accessibility + Automation; never Screen Recording
**Why:** titles are empty today because the current crate falls back to CGWindowList (needs Screen Recording). AX (Accessibility) gets titles with a lighter, less-scary permission; AppleScript/Automation gets exact browser URLs. Screen Recording is the worst look for a privacy app. **Evidence:** DB inspection showed ~363/370 Chrome rows and ~100% of Cursor/iTerm rows with empty titles. **Rejected:** Accessibility-only (weaker browser data), app-level-only (guts the product), Screen Recording (toxic optics).

### D5 ‚ÄĒ Event-driven capture + heartbeat
**Why:** NSWorkspace activation + AX focus-change observers give second-accurate boundaries and near-zero idle CPU; a slow heartbeat handles idle + long-running windows. **Rejected:** 5s polling (blurry boundaries, constant wakeups).

### D6 ‚ÄĒ Projects auto-inferred (+ correctable); needs a spike
**Why:** low effort, gets smarter. **Risk:** inference accuracy is unproven ‚ÄĒ **Phase 0 spike required** before committing the pattern. **Rejected:** manual rules (upkeep forever), defer-to-AI (loses the axis early).

### D7 ‚ÄĒ Pre-AI sorting: smart default rules + edits
**Why:** real value day one (VS Code‚ÜíDeep, Slack‚ÜíComms‚Ä¶), AI refines later. **Rejected:** blank cold-start, uncategorized-until-AI (kills the dial's colors).

### D8 ‚ÄĒ Sensitive data: raw-local + exclusion list
**Why:** all local, but a privacy product must let you exclude password managers/banking, mark private apps (time counts, no title), and never record incognito. **Rejected:** raw-no-filter (invasive on screen-share), categories-only (guts detail).

### D9 ‚ÄĒ Smart runtime: Apple Foundation Models via Swift sidecar; template fallback always
**Why:** on macOS 26 it's free, fully on-device, zero-install, structured output. Cost: a thin Swift bridge + Apple-Silicon/AI-on requirement (fallback covers the rest). **Rejected:** Ollama (install friction), embedded model (app bloat).

### D10 ‚ÄĒ Categorization via embeddings + corrections
**Why:** fast, stable, can't hallucinate a category; learns from each fix; NaturalLanguage gives on-device embeddings free. **Rejected:** nightly LLM classify (drift), rules+AI-suggestions only (too manual).

### D11 ‚ÄĒ Recap: lazy + opt-in evening ping; model narrates only
**Why:** computed on open keeps it an "open it to look" ritual; one optional bedtime notification is the sole sanctioned interruption. Numbers are computed in Rust; the model only phrases them. **Rejected:** never-notify (easy to forget), always-live (loses the end-of-day moment).

### D12 ‚ÄĒ On-device only for v1 (cloud is a future maybe)
**Why:** "nothing leaves this machine" is the moat and must be literally true + auditable. **Status:** BYO-key cloud is a deferred, opt-in-with-warnings possibility, not built.

### D13 ‚ÄĒ App shape: menubar launcher + main window
**Why:** always-running and one click to look, without a live HUD (rejected in D1). **Rejected:** popover-primary (cramps the dial), dock-only (easy to forget it runs).

### D14 ‚ÄĒ Dial: fixed 24h, midnight top; idle = faint hollow arcs
**Why:** consistent scale makes the week comparable (the whole point of the week view); the clock metaphor stays pure; empty arc = sleep is honest. A "day starts at 4am" offset is a later setting for night owls. **Rejected:** rolling 24h (breaks the clock + "your day"), auto-fit (every day looks identical ‚ÄĒ demonstrated in a mockup), per-user configurable scale (complexity).

### D15 ‚ÄĒ Linear timeline in v1 (secondary), dark mode at launch
**Why:** the timeline is the same event log rendered differently (cheap); dark mode is doable cleanly via token-based theming and many Mac users live in dark. **Note:** dark Bauhaus must be *designed*, not auto-derived ‚ÄĒ real design time.

### D16 ‚ÄĒ Architecture: Rust core (objc2) + thin Swift AI sidecar
**Why:** keep ~90% in one language; objc2 reaches NSWorkspace + AX from Rust directly; Swift is needed only for the Swift-only Foundation Models framework, kept to a minimal stdio sidecar. **Rejected:** bigger Swift core (splits codebase), pure-Rust-defer-AI (no smart recap).

### D17 ‚ÄĒ IPC: tauri-specta generated bindings
**Why:** the Rust‚ÜĒTS boundary is where drift hides; generating the client+types makes a shape mismatch a compile error. The biggest "trust code you didn't read" lever. **Rejected:** hand-written wrappers (silent desync), ts-rs types-only (half the win).

### D18 ‚ÄĒ Persistence: rusqlite + typed repository + integration tests
**Why:** tiny schema + analytics-heavy queries where ORMs are weak; simplest, most auditable, no async footguns for a single-writer local DB. **Rejected:** ORM (Diesel/SeaORM ‚ÄĒ wrong fit, heavy magic, you drop to raw SQL anyway), sqlx (compile-checked SQL is nice but async + a stale-able offline-prepare cache isn't worth it here). _Open to sqlx if compile-time SQL checking becomes a priority._ **Confirmed by reality:** `main` already uses rusqlite + a versioned migration system (`schema_migrations`) + tests ‚ÄĒ this matches what's built.

### D19 ‚ÄĒ Design system: Claude Design project + in-repo `design-system.md`, designed end-to-end first
**Why:** a living visual source of truth + a machine-readable spec the coder obeys, fully designed (all components, both themes, all states) **before** any UI code, to eliminate drift. **Dependency:** pushing to Claude Design needs the claude.ai design login / `/design-login`.

### D20 ‚ÄĒ Open source MIT day one; notarized direct distribution; free + sponsor
**Why:** auditability is the privacy proof; MAS sandbox forbids our permissions; free maximizes trust/adoption. **Rejected:** closed source (undercuts the claim), MAS (impossible), paid-upfront (wrong for OSS v1).

### D21 ‚ÄĒ Onboarding: primed first-run, run degraded
**Why:** explain *why* each permission, deep-link to Settings, never wall the app; degraded (app-level) if declined. **Rejected:** hard gate (first-launch wall), silent native prompts (scary, unexplained).

### D22 ‚ÄĒ First milestone: native capture spike (isolated) ‚Üí then the real dial
**Why:** prove the riskiest bit (AX titles + Automation URLs + NSWorkspace events) in isolation before product code, then wire to DB and render today's real dial. Design system proceeds as a parallel gating track.

### D23 ‚ÄĒ Discard the uncommitted XP/goals WIP ‚ÄĒ but build ON existing `main`
**Why:** the XP/goals/old-SettingsView work was uncommitted and off-direction ‚Üí dropped. **Correction:** an earlier draft said "start clean from scratch" ‚ÄĒ that was based on a stale local clone ~10 commits behind origin. The real `main` is a healthy, tested, CI'd v0.1.0 + OSS-hygiene foundation; we build on it (see D24).

### D24 ‚ÄĒ Evolve the existing codebase, don't rebuild
**Why:** `main` already has the Tauri shell, rusqlite + migrations, the rules engine, 24 Rust + 28 TS tests, CI, retention, and OSS docs ‚ÄĒ all worth keeping. The redesign changes: **capture** (`active_win_pos_rs` ‚Üí AX/Automation/NSWorkspace via objc2), the **data model** (new migrations for projects/sites/contexts/recaps/embeddings/exclusions), the **UI** (cyberpunk dashboard ‚Üí Bauhaus dial), and adds the **Swift AI sidecar + tauri-specta**. Keep the migration system, tests, CI, and repo hygiene.

### D25 ‚ÄĒ Follow the existing agent/PR workflow
**Why:** the repo runs a structured workflow (`context/plans/` with dated, task/workflow-numbered docs; PR-based; a headless-Linux build box that can't launch the desktop GUI). The redesign plan slots into it; capture/AX work is validated on macOS only. _Lesson: always fetch before assuming local state ‚ÄĒ this session started ~10 commits stale._

_D29 added 2026-06-22 from Spike ‚Ď° (`spikes/ax-observer/`)._

### D29 ‚ÄĒ Event-driven capture model: `objc2-application-services` `AXObserver` on the **main run loop**, marshaled to async over a `Send` channel
**Refines D5; proven by Spike ‚Ď°.** **Why:** the Phase-1 `capture` impl is now pinned to a model that was measured working end-to-end on the real Mac, Accessibility-only. The shape: a `NSWorkspace` `didActivateApplication` observer (a Rust closure in `block2::RcBlock`; **keep the returned token alive**) fires on each app switch; on switch we tear down the old per-PID `AXObserver` and build a fresh one (its run-loop source auto-detaches on release ‚ÄĒ remove it first to avoid a UAF against the live callback `refcon`); the observer watches `AXFocusedWindowChanged` on the **application** element, and `AXTitleChanged` on the **focused window** (re-pointed whenever the focused window changes, or in-app title changes never fire); chatty duplicate titles are dropped by a dedupe debounce; every event is `send`-marshaled to the existing Tokio runtime over an unbounded `Send` channel ‚ÄĒ a non-blocking enqueue, so the executor never stalls and the run loop never blocks. **Three findings that closed open questions:** (1) **`AXObserver` is exposed by `objc2-application-services` 0.3.2** (`AXObserver::create` / `add_notification` / `run_loop_source`) ‚ÄĒ so the whole capture surface stays in one objc2 family and **`accessibility-sys` is dropped** (corrects the architecture doc + the capture standard's "pick one" note); (2) **no `NSApplication` is needed** ‚ÄĒ a bare `CFRunLoop` delivers activation notifications, and in the app we register sources into **Tauri's main run loop** during `setup` rather than owning the loop; (3) AX queries + the observer source both run on the **main thread** (D5's "AX is main-thread" stands; the constraint is *where the source is serviced*, not that queries are main-only). **Rejected:** a dedicated `std::thread` running its own `CFRunLoopRun()` (unnecessary once the main-loop model was proven, and it would add cross-thread AX-safety questions); `accessibility-sys` for observers (a second FFI style for no gain). Embeddings/idle/etc. unaffected. _Residual: pure in-place `AXTitleChanged` delivery still to be confirmed in a manual browser-navigation pass; the production debounce may add a trailing-edge timer to also coalesce rapid distinct titles._

_D30 added 2026-06-22 from the project-inference spike (`spikes/project-inference/`)._

### D30 ‚ÄĒ Project identity is canonicalized on the git remote; assignment abstains below a confidence/ambiguity threshold
**Refines D6; informed by the project-inference spike.** **Why:** the spike measured the inference heuristic on real signals and made **zero false assignments**, but surfaced two design requirements. (1) **Canonical project identity = the git remote `owner/repo`.** The same project arrives under different ids depending on the signal ‚ÄĒ `cwd ‚Üí git remote` gives `f-gozie/usage-os`, while the window title / folder gives `usage_os`, and a `github.com/owner/repo` URL gives `usenudgeai/nudge` while a local file gives `nudge`. Without reconciliation **one project fragments into several**, so the `projects` table keys on the git remote and stores the folder name, title-derived name, and any GitHub URL as **aliases** that resolve to it (folder name is the fallback key when a repo has no remote). (2) **An explicit abstain threshold** (the calm-mirror principle: a wrong project label costs more trust than a gap): assign a project only on a HIGH (`cwd-git-remote`, `github-url`) or MED (`local-file`, `window-title`) **and unambiguous** signal; otherwise persist **`unassigned`**. Signal-precision order: `cwd ‚Üí git remote` is the anchor (deterministic, canonical, no false positives), corroborated by `github-url` (R26); window-title is the weakest (folder-name only). (3) **`ambiguous` is a distinct third state** from `no-signal`: `localhost` and dev dashboards (PostHog, Grafana, Cloudflare, App Store Connect) are clearly *work* but project-unknown ‚ÄĒ abstain now, but **persist the ambiguous reason** so Phase 2 can temporally **correlate** them to the project active in the editor/terminal at the same time (never correlate `no-signal` browsing like YouTube/X/Gmail). **Rejected:** folder name as the canonical key (fragments across renames/forks/worktrees); guessing a project for ambiguous tooling (the main false-positive source); a single binary "has project / no project" flag (loses the correlate-later signal). _Measured on a snapshot ‚ÄĒ precision + the abstain threshold; multi-day **recall**/volume is re-measured once Phase-1 capture persists data (R23/R26/R27)._

### D32 ‚ÄĒ Capture is split into the seam (1.2a, CI-tested) and the macOS impl (1.2b, on-device)
**Implements hard rule 5 + D22/D29.** **Why:** the event-driven macOS capture is the riskiest native code and can only be **runtime-verified on a real Mac** (TCC + GUI) ‚ÄĒ CI compiles it but cannot exercise it. So capture ships in two PRs: **1.2a** builds the whole CI-testable spine, **1.2b** drops in the native impl behind the same trait, verified on-device. This keeps the gnarly FFI isolated for separate iteration and never leaves `main` un-CI-able. **The seam (1.2a):** a `capture/` module with a `CaptureSource` trait (`fn start(self: Box<Self>, tx)`), a `FocusEvent` (owned/`Send`, marshaled per D29), a `process_focus_event` consumer that applies D8 exclusions (`Exclude` drops, `Private` records time but omits title/url) then coalesce-writes via `db::log_focus`, a `run` runner draining the channel, and the health counter behind `get_watcher_status`. Two impls today: `FakeCapture` (seeded events ‚ÄĒ the mock that lets the spine run headless, hard rule 5 / C4) and `PollingCapture` (the v0.1.0 `active-win-pos-rs` + `user-idle` logic moved behind the trait ‚ÄĒ the interim production source so the app never regresses; kept afterwards as the non-macOS fallback). `default_source()` is the single switch 1.2b flips to the macOS impl. **`watcher.rs` is deleted**, its role folded into `capture::run`; the IPC command keeps the name `WatcherStatus`/`get_watcher_status` (renaming would churn `bindings.ts`). **Deliberate behavior note:** `PollingCapture` captures **app name + idle only ‚ÄĒ no window titles** (it must not read `active-win-pos-rs` titles, which come from `CGWindowName`/Screen Recording ‚ÄĒ capture standard C1). Real titles (+ url/cwd) arrive only with the AX impl in 1.2b; until then the dial shows apps, honestly, rather than the empty titles the v0.1.0 path produced. **1.2b scope:** NSWorkspace + per-PID `AXObserver` on Tauri's main run loop (port `spikes/ax-observer`), browser URL + incognito skip (port `spikes/browser-url`, C5/C6), terminal cwd + front-tab pid selection (port `spikes/proc-cwd`), then site parsing + project inference (port `spikes/project-inference` ‚Üí `resolve_or_create_project`). **No IPC change in 1.2a** (`bindings.ts` byte-identical). **Rejected:** one big PR with the native impl (CI can't verify the native half ‚ÄĒ worse review + on-device coupling); ripping out the polling watcher before the AX impl exists (capture gap)._

_D33 added 2026-06-23 from Phase 1.2b (`phase1/capture-macos`)._

### D33 ‚ÄĒ Event-driven macOS capture: ported the spikes behind the trait; enrichment inline; dedicated consumer thread
**Implements D29/D30 + capture standard C1/C5/C10; completes D32's 1.2b.** **Why:** drop the real capture into the seam so the app gets **event-driven app + window title + browser URL + terminal project** instead of polled app-names. The native code (`capture/macos/{mod,browser,terminal}.rs`, `#[cfg(target_os="macos")]`) is a near-verbatim port of the four proven spikes; it **compiles + clippy-passes on macOS CI** but its runtime is **verified on-device** (TCC + GUI), so the PR carries an on-device checklist rather than CI assertions. **Shape:** `MacosCapture::start` runs on the main thread (Tauri `setup`), registers the NSWorkspace activation block + per-PID `AXObserver` on `CFRunLoop::current()`, and **leaks** the `!Send` keep-alives (`mem::forget`) since capture runs for the whole process and can't live in `Send` Tauri state. **Three design choices beyond the spikes:** (1) **enrichment is inline at capture, not a later batch pass** ‚ÄĒ the terminal cwd signal is ephemeral, so `process_focus_event` resolves site (`enrich::parse_site`) + project (`enrich::infer_project`, the ported heuristic) immediately, persisting `project_id` or the abstain *kind*; project identity reconciles weak signals (folder/title/file names) to an existing project via a `"name"` alias so one project never fragments (D30). (2) **The capture channel moved from `tokio::mpsc` to `std::mpsc` + a dedicated consumer `std::thread`** (R57 direction) because per-event processing now blocks on SQLite **and** a `git` shell (inference) ‚ÄĒ neither may run on the async executor; `capture::run(db_conn, source)` split into `start` (main thread) + `consume` (dedicated thread). (3) **Terminal front-tab pid selection has no clean API** ‚ÄĒ iTerm2 is read directly via AppleScript `session.path`; other terminals fall back to a best-effort heuristic (highest-pid child with a readable cwd via `pgrep -P` + `proc_pidinfo`), abstaining (‚Üí `unassigned`, degrades cleanly) when unresolved. **Safari URL is intentionally not read** (no scriptable private-window property ‚Üí D8 safe-default, R18). **Deferred:** the heartbeat timer (C2 ‚ÄĒ bounding long idle spans needs a main-run-loop `CFRunLoopTimer`; idle is currently sampled per event), and full permission priming/onboarding (Phase 4 / D21 ‚ÄĒ for now an untrusted AX state prompts once and degrades to no-titles). **No IPC change** (`bindings.ts` byte-identical). objc2 deps unified cleanly with Tauri's (0.6.4 / 0.3.2 train). **Rejected:** project inference as a batch pass (cwd is ephemeral ‚ÄĒ must resolve live); storing raw cwd on the event (no column, and it's transient); reading Safari URLs without proven non-private state (D8 risk)._

### D35 ‚ÄĒ Migrations: hand-written per-file SQL on a hardened in-house runner; clean-slate baseline squash
**Revisits D31's "append-only v5‚Äďv8" now that the UI rewrite touches the schema again, prompted by the question "is the inline `const MIGRATIONS` array the right pattern for an OSS project?"** **Why:** the old runner had real problems for a public, contributable codebase ‚ÄĒ (1) a **correctness bug**: each migration's DDL and its `schema_migrations` bookkeeping ran **un-transacted**, so a mid-migration failure left a half-applied, *unrecorded* migration ‚Üí next boot re-runs a non-idempotent `ALTER` ‚Üí "duplicate column", app won't start; (2) **all migrations inline in the 1.6k-line `db.rs`** ‚Üí every migration PR edits the same array (merge conflicts) and reviewing a migration means scrolling a giant file; (3) **SQL as Rust string literals** (no highlighting/linting); (4) **no drift detection** ‚ÄĒ nothing stopped a contributor editing an already-shipped migration, silently diverging new installs from existing ones. **The decision (Path A ‚ÄĒ harden in-house, zero new deps):** keep **hand-written SQL** (SQLite's deliberately-limited `ALTER` makes auto-derived/ORM DDL *less* correct here, and hard rule 4 + the audit-the-source ethos reward few legible deps), but move it to **per-file `.sql` under `src-tauri/migrations/`** (`include_str!`'d), run by a top-level `migrations` module that applies **each migration in a transaction** (SQLite has transactional DDL ‚Üí atomic), stores a **stable FNV-1a checksum** per applied migration and **verifies it on every boot** (editing a shipped migration is now a loud startup error, not silent drift), and is **forward-only by design** (no down-migrations ‚ÄĒ a local-first desktop DB only rolls forward; documented in `migrations/README.md`). **Clean-slate squash:** since nothing has shipped, the historical v1‚Äďv8 chain (incremental ALTERs that only existed for installs that no longer exist) is **collapsed into one readable baseline** `0001_initial_schema.sql` (every column inline, including the new contexts `slug`), plus `0002_seed_canonical_contexts.sql`. No baseline cutover; existing dev DBs are throwaway capture-test data (delete before running the new build). **Contexts for the dial:** the internal table name stays `categories` (D31 rename still deferred), but a `slug` column (`deep|research|comms|breaks`) now carries the **canonical context identity the UI maps to a colour token** (`--c-<slug>`, theme-aware); the 4 canonical contexts are **seeded** (NULL slug = a user-created context with its own hex `color`). **Rejected:** the `refinery` crate (excellent ‚ÄĒ gives checksums + transactions for free ‚ÄĒ but adds a dependency and a fiddly history-table baseline cutover, neither worth it for ~80 reviewable lines on a deps-conscious project); auto-derived/ORM migrations (wrong for SQLite's ALTER limits + against hard rule 4); preserving the v1‚Äďv8 history (no value pre-1.0 ‚ÄĒ a squashed baseline reads far better for contributors). _No IPC change (`bindings.ts` byte-identical ‚ÄĒ `Category`/`get_categories` don't yet expose `slug`; the rollup reads it directly). ### D38 ‚ÄĒ Capture write-path: a single-writer in-memory state machine (replaces the two-thread "re-derive the last row" design)
**Supersedes the D37 mechanism (the close-on-switch + separate heartbeat thread shipped as the emergency fix). Prompted by "is this the best/simplest/most-efficient way?" and pressure-tested by two independent senior reviewers (an external Codex agent + an Opus agent) who converged.** **Why:** the D37 fix worked but had three overlapping mechanisms (coalesce / close-on-switch / heartbeat), **two threads both writing SQLite and both re-deriving "the current span" from `ORDER BY id DESC`**, a staleness heuristic, and ~6 magic constants ‚ÄĒ complexity that exists *only because* no one owned the open span. **The decision:** the consumer thread becomes the **sole DB writer** and owns the open span in memory (`SpanState { current: Option<OpenSpan>, last_focus: Option<Focus> }`); it **self-ticks via `Receiver::recv_timeout`** (the separate heartbeat thread is deleted) and the DB layer drops to dumb primitives. `db.rs` lost `log_focus`, `log_activity`/`log_activity_safe`, `extend_current_span`, `get_last_activity_log`, and the three gap/stale constants; `update_last_activity_end_time`‚Üí`set_span_end(id,end)`. Net: **two knobs** (`TICK_SECS=20`, `GATE_SECS=120`), one writer, one source of truth, the span coalescing/idle behaviour unit-tested in `capture` (no DB "last row" guessing). **The one bug both reviewers caught:** a naive `recv_timeout` tick only fires on channel *silence*, so a chatty live-title tab (CI dashboard, "(3) Slack", `npm run dev` rebuilds) while you're away starves the idle gate and balloons the span ‚ÄĒ so idle is evaluated on a **wall-clock deadline checked on every loop wake**, not by the timeout firing. **Resolved sub-decisions (where the reviewers initially split):** **(A)** keep `last_focus` and **reopen the same window after an idle-close on an active tick** ‚ÄĒ macOS fires no event when you return to an unchanged window, so without it "read ‚Üí pause ‚Üí keep reading the same doc" (the core deep-work case) silently undercounts. **(B)** extend the span to **`now`** (gate-checked, forward-only), *not* `now ‚ąí idle` ‚ÄĒ the bounded ‚Č§GATE tail is sub-perceptual for a rear-view tool and avoids two-time-source arithmetic. **(C)** `TICK=20s` (calm app; ~negligible vs 15) and `GATE=120s` (tie-break to the *honest-mirror* principle: over-counting erodes trust more than a modest under-count, and scroll/trackpad reset idle so genuine reading rarely trips it). **Free wins from single-writer:** startup `current=None` (no dangling-span recovery ‚ÄĒ the last row keeps its last-written end), extend-forward-only (handles sleep/wake + backward clock skew), and the per-event `SELECT last` disappears (fewer writes + shorter lock holds than D37). **Rejected:** tick-only sampling (discards the precise AX switch boundary, blurs short focuses, still needs idle policy); pure read-time duration reconstruction (idle/away is a capture-time fact ‚ÄĒ at read time you've lost "was input flowing"); a writer-actor/channel abstraction or a persisted open-span table (over-engineering at this scale); media/"is-playing" probes or a third threshold (keep two knobs; document the "no input = away after GATE" limitation). _Idle is now modelled as the *absence* of a span (untracked gap), not recorded idle rows. No IPC change (`bindings.ts` byte-identical). 69 Rust + 7 TS tests, clippy/fmt clean. Idle thresholds remain D34a dogfood-tunable; GATE is the one to watch._
