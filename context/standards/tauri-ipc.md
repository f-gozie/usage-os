# Standard: Generated IPC with tauri-specta

_Scope: the Rust‚ÜîTS boundary. How commands, types, errors, and events cross from the Rust core to the React frontend, and how that boundary is kept honest in CI._

_Status: **provisional**. Grounded in a single research pass with **no independent verification**. Version numbers and exact API signatures are treated as provisional until the Phase-0 spike confirms them against `Cargo.lock` and `docs.rs` for the pinned release-candidate. Anything not confirmed by a cited authoritative source lives under [Open questions](#-open-questions--verify-in-the-phase-0-spike), not in the asserted body._

---

## Why this exists (the rules this enforces)

Two of the project's hard rules (`CLAUDE.md`) live or die here:

> **2. The IPC contract is generated, never hand-written.** All Rust‚ÜîTS types/clients come from `tauri-specta`. Editing generated bindings by hand is forbidden.

> **8. Every merge is gated:** `cargo clippy -D warnings`, `cargo fmt --check`, `cargo test`, `tsc`, and a **binding-freshness check** must pass. Red = not merged.

And it touches a third indirectly:

> **3. No `unwrap()` / `expect()` / `panic!` in production paths.** Errors are typed and propagated (`Result`).

The architecture doc states the boundary plainly: _"Frontend ‚Üí Rust: only via the generated client. No fetch, no direct DB, no business logic in the UI."_ This standard is how we make "only via the generated client" a **compile-time** fact rather than a hope.

This is also decision **D17** ("Generated IPC with tauri-specta"). The exact release-candidate we pin must be recorded in `context/decisions.md` so versions do not float.

## The shape (what good looks like)

```
React component
   ‚îÇ  imports generated wrapper fn from src/bindings.ts  (typed, never hand-edited)
   ‚ñº
commands::get_categories   #[tauri::command] #[specta::specta]
   ‚îÇ  fn(args: typed) -> Result<T: specta::Type, AppError: specta::Type>
   ‚ñº
domain / store             (numbers computed here ‚Äî rule 6)
```

- One Rust command = one generated TS function with exact argument and return types.
- Every type that crosses the boundary derives **both** `serde::Serialize`/`Deserialize` **and** `specta::Type`.
- Errors cross as a typed `AppError` enum, not `String`.
- The generated `src/bindings.ts` is committed and regenerated by a test; CI fails if it drifts.

## Pattern 1 ‚Äî derive `Type` on every boundary type

Every struct/enum that appears in a command argument or return type needs `specta::Type` **in addition to** the serde derives it already has. The repo's `db.rs` structs (`ActivityLog`, `Category`, `Rule`) currently derive only `Serialize`/`Deserialize`.

```rust
use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, Serialize, Deserialize, specta::Type)]
pub struct Category {
    pub id: i64,
    pub name: String,
    pub color: String,
}
```

Rationale: a type the frontend touches but that lacks `specta::Type` simply won't generate ‚Äî the contract would have a hole.

---

## Pattern 2 ‚Äî annotate every command

Keep the existing `#[tauri::command]` and add `#[specta::specta]`. Tauri-injected arguments (e.g. `State<DbState>`) are skipped from the generated TS signature.

```rust
#[tauri::command]
#[specta::specta]
pub fn get_categories(db: tauri::State<DbState>) -> Result<Vec<Category>, AppError> {
    // delegates to the store layer ‚Äî no SQL here (rule 4)
    db.repo().categories().map_err(AppError::from)
}
```

Rationale: the attribute is what registers the command's full type signature with the generator; without it the command is invisible to the bindings.

---

## Pattern 3 ‚Äî typed errors, not `String`

All current commands return `Result<T, String>`. `String` exports (to TS `string`) so it _works_, but it defeats D17's intent ‚Äî a typed error contract. Replace it with a `thiserror` enum that derives `serde::Serialize` + `specta::Type`.

```rust
#[derive(Debug, thiserror::Error, serde::Serialize, specta::Type)]
#[serde(tag = "kind", content = "message")]
pub enum AppError {
    #[error("database error: {0}")]
    Db(String),
    #[error("not found")]
    NotFound,
    #[error("permission denied")]
    PermissionDenied,
}
```

Rationale: a typed error enum lets the frontend `switch` on `kind` instead of string-matching prose, and makes the failure modes part of the generated contract.

> Whether `thiserror` + `specta::Type` co-derive cleanly (no derive/orphan conflict) is **unverified** ‚Äî confirm in the spike.

## Pattern 4 ‚Äî no `serde_json::Value`, no bare tuples

`serde_json::Value` does **not** implement `specta::Type` (it is dynamically shaped). Any command returning it will **fail to generate**. The current `get_watcher_status` returns `Result<serde_json::Value, String>` and must be rewritten to a named struct:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, specta::Type)]
pub struct WatcherStatus {
    pub consecutive_errors: u32,
    pub healthy: bool,
}
```

`get_settings` returns `Vec<(String, String)>`, which exports as the awkward TS `[string, string][]`. Prefer a named struct for the settings shape.

Rationale: named structs generate stable, ergonomic, self-documenting TS; dynamic JSON and tuples either break generation or produce types nobody wants to consume.

---

## Pattern 5 ‚Äî `i64` timestamps cross as `number`, explicitly

By default `i64`/`u64` map to TS `bigint` (lossless but JSON-incompatible and painful for the dial/recap math). UsageOS uses Unix-**second** timestamps, far below 2^53, so coercing to `number` is safe. Set this explicitly on the exporter ‚Äî the default differs.

```rust
specta_typescript::Typescript::default()
    .bigint(specta_typescript::BigIntExportBehavior::Number)
```

Rationale: epoch-second `i64`s are well within `Number.MAX_SAFE_INTEGER`, so `Number` keeps the TS dial/recap arithmetic ergonomic without precision loss.

## Pattern 6 ‚Äî the Builder wiring (replaces `generate_handler!`)

The v2 API is Builder-based. The hand-written `tauri::generate_handler![...]` in `src-tauri/src/lib.rs` must be **fully replaced** by `builder.invoke_handler()` ‚Äî keeping both registers commands twice.

```rust
// src-tauri/src/lib.rs  ‚Äî PROVISIONAL, confirm method signatures on docs.rs for the pinned rc
fn make_builder() -> tauri_specta::Builder<tauri::Wry> {
    tauri_specta::Builder::<tauri::Wry>::new()
        .commands(tauri_specta::collect_commands![
            crate::commands::get_categories,
            crate::commands::create_category,
            crate::commands::get_watcher_status,
            // ... all commands listed here
        ])
        .events(tauri_specta::collect_events![]) // empty until events are de-risked (Pattern 7)
}

pub fn run() {
    let builder = make_builder();

    tauri::Builder::default()
        .invoke_handler(builder.invoke_handler())
        .setup(move |app| {
            builder.mount_events(app); // only needed once events exist
            Ok(())
        })
        .run(tauri::generate_context!())
        .expect("error while running tauri application");
}
```

Rationale: a single source of command registration (the Builder) is what guarantees the generated TS and the runtime handler can never disagree.

> The exact method names/signatures (`invoke_handler()`, `mount_events(app)`) are reconstructed from secondary sources, not quoted from primary docs ‚Äî confirm in the spike.

## Pattern 7 ‚Äî events are deferred, not assumed

v2 supports typed events (`collect_events!` + `mount_events` + a generated typed `listen`), and event payload structs must also derive `specta::Type`. **However**, there is a known open bug ‚Äî tauri-specta issue **#211** ‚Äî where on `rc.24` the generated `makeEvent` payload type can infer to `never`, breaking typed emit on the TS side.

Position for launch: **ship commands-only.** The recap and dial are **pull-based** (the frontend invokes, Rust returns), so typed events are not on the critical path. Keep `.events(collect_events![])` empty until a spike proves the payload type is the struct and not `never`.

Rationale: the lowest-risk boundary that satisfies the product is commands-only; events buy nothing the pull-based UI needs and currently carry a known correctness bug on the pinned RC.

## Pattern 8 ‚Äî export bindings from a test, never from a production path

The export call uses `.expect()`/`.unwrap()` in every example. That is **fine here** because it lives in a `#[test]` / debug-only codegen path, **not** a production path ‚Äî so it does **not** violate hard rule 3. Keep it out of `run()`.

```rust
// src-tauri/src/export_bindings.rs (or #[cfg(test)] module)
#[test]
fn export_bindings() {
    crate::make_builder()
        .export(
            specta_typescript::Typescript::default()
                .bigint(specta_typescript::BigIntExportBehavior::Number)
                .header("// @generated by tauri-specta. DO NOT EDIT.\n"),
            "../src/bindings.ts",
        )
        .expect("failed to export bindings");
}
```

Two non-negotiables in that snippet:

1. **The header banner** (`@generated ... DO NOT EDIT`) is mandatory ‚Äî it is how reviewers and CI detect hand edits (hard rule 2).
2. **The formatter must be deterministic and identical on every runner**, or the freshness gate produces spurious diffs. Prefer **biome** (a Rust binary ‚Äî no Node version skew) or pin **prettier** exactly (Node 22 is already in CI).

Rationale: codegen is not a production code path, so an `expect()` there is allowed; the banner + deterministic formatter are what make the freshness gate trustworthy.

---

## Pattern 9 ‚Äî the binding-freshness CI gate (hard rule 8)

The committed `src/bindings.ts` must match what the export test produces. The gate: run the export, then assert no git diff. Run it **before** `tsc`.

```yaml
# .github/workflows ‚Äî extend the existing CI, do not replace it
- name: Regenerate IPC bindings
  run: cargo test --manifest-path src-tauri/Cargo.toml export_bindings
- name: Assert bindings are fresh
  run: git diff --exit-code -- src/bindings.ts
```

This runs on the headless **Linux** runner ‚Äî fine, because the export is pure codegen (no macOS APIs). The only failure mode to defend against is formatter non-determinism (Pattern 8).

Rationale: "generated, never hand-written" is only real if a green build is impossible whenever the committed file and the regenerated file disagree.
