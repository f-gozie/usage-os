# Standard: Generated IPC with tauri-specta

_Scope: the Rust↔TS boundary. How commands, types, errors, and events cross from the Rust core to the React frontend, and how that boundary is kept honest in CI._

_Status: **provisional**. Grounded in a single research pass with **no independent verification**. Version numbers and exact API signatures are treated as provisional until the Phase-0 spike confirms them against `Cargo.lock` and `docs.rs` for the pinned release-candidate. Anything not confirmed by a cited authoritative source lives under [Open questions](#-open-questions--verify-in-the-phase-0-spike), not in the asserted body._

---

## Why this exists (the rules this enforces)

Two of the project's hard rules (`CLAUDE.md`) live or die here:

> **2. The IPC contract is generated, never hand-written.** All Rust↔TS types/clients come from `tauri-specta`. Editing generated bindings by hand is forbidden.

> **8. Every merge is gated:** `cargo clippy -D warnings`, `cargo fmt --check`, `cargo test`, `tsc`, and a **binding-freshness check** must pass. Red = not merged.

And it touches a third indirectly:

> **3. No `unwrap()` / `expect()` / `panic!` in production paths.** Errors are typed and propagated (`Result`).

The architecture doc states the boundary plainly: _"Frontend → Rust: only via the generated client. No fetch, no direct DB, no business logic in the UI."_ This standard is how we make "only via the generated client" a **compile-time** fact rather than a hope.

This is also decision **D17** ("Generated IPC with tauri-specta"). The exact release-candidate we pin must be recorded in `context/decisions.md` so versions do not float.

---

## The shape (what good looks like)

```
React component
   │  imports generated wrapper fn from src/bindings.ts  (typed, never hand-edited)
   ▼
commands::get_categories   #[tauri::command] #[specta::specta]
   │  fn(args: typed) -> Result<T: specta::Type, AppError: specta::Type>
   ▼
domain / store             (numbers computed here — rule 6)
```

- One Rust command = one generated TS function with exact argument and return types.
- Every type that crosses the boundary derives **both** `serde::Serialize`/`Deserialize` **and** `specta::Type`.
- Errors cross as a typed `AppError` enum, not `String`.
- The generated `src/bindings.ts` is committed and regenerated by a test; CI fails if it drifts.

## Versions — pin exactly, move as a set

The whole specta v2 stack is **release-candidate** (`specta` v2 has had no stable release since the 2022 1.x line). Maintainers advise locking with `=` because RC releases carry breaking changes. Pin the trio and move them together; never float.

```toml
# src-tauri/Cargo.toml  — PROVISIONAL pins, confirm against Cargo.lock in the spike
[dependencies]
tauri = { version = "2", features = [] }
tauri-specta = { version = "=2.0.0-rc.24", features = ["derive", "typescript"] }
specta = "=2.0.0-rc.24"
specta-typescript = "=0.0.11"
# optional, for thiserror-based AppError:
thiserror = "2"
```

| Crate | Pin (provisional) | Rationale |
|---|---|---|
| `tauri-specta` | `=2.0.0-rc.24` | Typesafe Tauri v2 commands. `rc.24` is the version the docs/issues reference; latest available is `rc.25`. Exact-pin: RC API can break. |
| `specta` | `=2.0.0-rc.24` | Provides the `Type` trait/derive. Must match `tauri-specta`'s RC or builds break. |
| `specta-typescript` | `=0.0.11` | The TS exporter (`Typescript`, `BigIntExportBehavior`, formatter). Versioned independently; `0.0.11` pairs with the `rc.24` trio. |
| `tauri` | `2.x` (repo already on `"2"`) | `tauri-specta` v2 targets Tauri v2. No conflict expected; confirm in spike. |

Rationale (one line): the stack is pre-1.0 RC, so an exact, recorded pin is the only thing that keeps a `cargo update` from silently breaking the IPC contract.

> rc.24 vs rc.25, and the exact `specta` version `tauri-specta` actually resolves to, are **open questions** — see below. Do not assert a pin in `decisions.md` until the spike reads it back from `Cargo.lock`.

## Pattern 1 — derive `Type` on every boundary type

Every struct/enum that appears in a command argument or return type needs `specta::Type` **in addition to** the serde derives it already has. The repo's `db.rs` structs (`ActivityLog`, `Category`, `Rule`) currently derive only `Serialize`/`Deserialize`.

```rust
use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, Serialize, Deserialize, specta::Type)]
pub struct Category {
    pub id: i64,
    pub name: String,
    pub color: String,
}
```

Rationale: a type the frontend touches but that lacks `specta::Type` simply won't generate — the contract would have a hole.

---

## Pattern 2 — annotate every command

Keep the existing `#[tauri::command]` and add `#[specta::specta]`. Tauri-injected arguments (e.g. `State<DbState>`) are skipped from the generated TS signature.

```rust
#[tauri::command]
#[specta::specta]
pub fn get_categories(db: tauri::State<DbState>) -> Result<Vec<Category>, AppError> {
    // delegates to the store layer — no SQL here (rule 4)
    db.repo().categories().map_err(AppError::from)
}
```

Rationale: the attribute is what registers the command's full type signature with the generator; without it the command is invisible to the bindings.

---

## Pattern 3 — typed errors, not `String`

All current commands return `Result<T, String>`. `String` exports (to TS `string`) so it _works_, but it defeats D17's intent — a typed error contract. Replace it with a `thiserror` enum that derives `serde::Serialize` + `specta::Type`.

```rust
#[derive(Debug, thiserror::Error, serde::Serialize, specta::Type)]
#[serde(tag = "kind", content = "message")]
pub enum AppError {
    #[error("database error: {0}")]
    Db(String),
    #[error("not found")]
    NotFound,
    #[error("permission denied")]
    PermissionDenied,
}
```

Rationale: a typed error enum lets the frontend `switch` on `kind` instead of string-matching prose, and makes the failure modes part of the generated contract.

> Whether `thiserror` + `specta::Type` co-derive cleanly (no derive/orphan conflict) is **unverified** — confirm in the spike.

---

## Pattern 4 — no `serde_json::Value`, no bare tuples

`serde_json::Value` does **not** implement `specta::Type` (it is dynamically shaped). Any command returning it will **fail to generate**. The current `get_watcher_status` returns `Result<serde_json::Value, String>` and must be rewritten to a named struct:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, specta::Type)]
pub struct WatcherStatus {
    pub consecutive_errors: u32,
    pub healthy: bool,
}
```

`get_settings` returns `Vec<(String, String)>`, which exports as the awkward TS `[string, string][]`. Prefer a named struct for the settings shape.

Rationale: named structs generate stable, ergonomic, self-documenting TS; dynamic JSON and tuples either break generation or produce types nobody wants to consume.

---

## Pattern 5 — `i64` timestamps cross as `number`, explicitly

By default `i64`/`u64` map to TS `bigint` (lossless but JSON-incompatible and painful for the dial/recap math). UsageOS uses Unix-**second** timestamps, far below 2^53, so coercing to `number` is safe. Set this explicitly on the exporter — the default differs.

```rust
specta_typescript::Typescript::default()
    .bigint(specta_typescript::BigIntExportBehavior::Number)
```

Rationale: epoch-second `i64`s are well within `Number.MAX_SAFE_INTEGER`, so `Number` keeps the TS dial/recap arithmetic ergonomic without precision loss.

---

## Pattern 6 — the Builder wiring (replaces `generate_handler!`)

The v2 API is Builder-based. The hand-written `tauri::generate_handler![...]` in `src-tauri/src/lib.rs` must be **fully replaced** by `builder.invoke_handler()` — keeping both registers commands twice.

```rust
// src-tauri/src/lib.rs  — PROVISIONAL, confirm method signatures on docs.rs for the pinned rc
fn make_builder() -> tauri_specta::Builder<tauri::Wry> {
    tauri_specta::Builder::<tauri::Wry>::new()
        .commands(tauri_specta::collect_commands![
            crate::commands::get_categories,
            crate::commands::create_category,
            crate::commands::get_watcher_status,
            // ... all commands listed here
        ])
        .events(tauri_specta::collect_events![]) // empty until events are de-risked (Pattern 7)
}

pub fn run() {
    let builder = make_builder();

    tauri::Builder::default()
        .invoke_handler(builder.invoke_handler())
        .setup(move |app| {
            builder.mount_events(app); // only needed once events exist
            Ok(())
        })
        .run(tauri::generate_context!())
        .expect("error while running tauri application");
}
```

Rationale: a single source of command registration (the Builder) is what guarantees the generated TS and the runtime handler can never disagree.

> The exact method names/signatures (`invoke_handler()`, `mount_events(app)`) are reconstructed from secondary sources, not quoted from primary docs — confirm in the spike.

## Pattern 7 — events are deferred, not assumed

v2 supports typed events (`collect_events!` + `mount_events` + a generated typed `listen`), and event payload structs must also derive `specta::Type`. **However**, there is a known open bug — tauri-specta issue **#211** — where on `rc.24` the generated `makeEvent` payload type can infer to `never`, breaking typed emit on the TS side.

Position for launch: **ship commands-only.** The recap and dial are **pull-based** (the frontend invokes, Rust returns), so typed events are not on the critical path. Keep `.events(collect_events![])` empty until a spike proves the payload type is the struct and not `never`.

Rationale: the lowest-risk boundary that satisfies the product is commands-only; events buy nothing the pull-based UI needs and currently carry a known correctness bug on the pinned RC.

---

## Pattern 8 — export bindings from a test, never from a production path

The export call uses `.expect()`/`.unwrap()` in every example. That is **fine here** because it lives in a `#[test]` / debug-only codegen path, **not** a production path — so it does **not** violate hard rule 3. Keep it out of `run()`.

```rust
// src-tauri/src/export_bindings.rs (or #[cfg(test)] module)
#[test]
fn export_bindings() {
    crate::make_builder()
        .export(
            specta_typescript::Typescript::default()
                .bigint(specta_typescript::BigIntExportBehavior::Number)
                .header("// @generated by tauri-specta. DO NOT EDIT.\n"),
            "../src/bindings.ts",
        )
        .expect("failed to export bindings");
}
```

Two non-negotiables in that snippet:

1. **The header banner** (`@generated ... DO NOT EDIT`) is mandatory — it is how reviewers and CI detect hand edits (hard rule 2).
2. **The formatter must be deterministic and identical on every runner**, or the freshness gate produces spurious diffs. Prefer **biome** (a Rust binary — no Node version skew) or pin **prettier** exactly (Node 22 is already in CI).

Rationale: codegen is not a production code path, so an `expect()` there is allowed; the banner + deterministic formatter are what make the freshness gate trustworthy.

---

## Pattern 9 — the binding-freshness CI gate (hard rule 8)

The committed `src/bindings.ts` must match what the export test produces. The gate: run the export, then assert no git diff. Run it **before** `tsc`.

```yaml
# .github/workflows — extend the existing CI, do not replace it
- name: Regenerate IPC bindings
  run: cargo test --manifest-path src-tauri/Cargo.toml export_bindings
- name: Assert bindings are fresh
  run: git diff --exit-code -- src/bindings.ts
```

This runs on the headless **Linux** runner — fine, because the export is pure codegen (no macOS APIs). The only failure mode to defend against is formatter non-determinism (Pattern 8).

Rationale: "generated, never hand-written" is only real if a green build is impossible whenever the committed file and the regenerated file disagree.

## Migration checklist (the current repo → generated IPC)

The repo currently wires commands by hand. This is mechanical but repo-wide.

- [ ] Add the pinned trio to `src-tauri/Cargo.toml`; record the chosen rc in `context/decisions.md` (D17).
- [ ] Add `#[derive(specta::Type)]` to every boundary struct (`ActivityLog`, `Category`, `Rule`, …).
- [ ] Add `#[specta::specta]` to all 11 existing commands.
- [ ] Define `AppError` (thiserror + `Serialize` + `Type`); replace `Result<_, String>` returns.
- [ ] Rewrite `get_watcher_status` (`serde_json::Value` → `WatcherStatus` struct).
- [ ] Rewrite `get_settings` (`Vec<(String,String)>` → named struct).
- [ ] Replace `tauri::generate_handler![...]` with the `Builder` in `lib.rs`.
- [ ] Add the export `#[test]` with `BigIntExportBehavior::Number` + the DO-NOT-EDIT header.
- [ ] Commit the generated `src/bindings.ts`; import it from the frontend.
- [ ] Add the freshness gate to CI; prove a no-op run yields zero diff on Linux.
