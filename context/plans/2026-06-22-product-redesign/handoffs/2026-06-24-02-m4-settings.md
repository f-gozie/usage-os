## 8. Important context / gotchas
- **The rename is boundary-only by design (D31/D42).** SQL keeps `categories`/`category_id`; `ActivityLog.category_id` keeps the legacy field name (hot path). The `Context`/`Rule.context_id` asymmetry is intentional and documented in `db.rs`.
- **Editing a canonical context's colour is a no-op visually** — the dial maps canonical `slug`→theme token (`contextColorVar`), not the stored hex. So the editor **hides the colour picker for the canonical four** (rename + rules only) and only shows swatches for user contexts. Don't "fix" this by reading the stored colour in the dial.
- **`reprocessLogs()` runs after every context/rule save** — it's the recategorize step; it resets `activity_logs.category_id` and re-applies all rules (first-match-wins, rule id order). Exclusion edits do **not** reprocess (they're capture-time only; existing data is unaffected).
- **Export writes next to the DB** (`~/Library/Application Support/com.favour.usage-os/usageos-export-<ts>.csv`) and reveals via `opener` `revealItemInDir` — needed the `opener:allow-reveal-item-in-dir` capability.
- **Delete-all preserves config** (contexts/rules/exclusions/settings/migrations) and wipes only the captured record (events + derived projects/sites). No VACUUM.
- **Bindings:** any backend command/type change requires `cargo test export_bindings` + committing `src/bindings.ts` (CI freshness gate) and updating `src/lib/tauri.ts` in lockstep or `tsc` fails.
