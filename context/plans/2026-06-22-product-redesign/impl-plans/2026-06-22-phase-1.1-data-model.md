## What landed

**Migrations v5–v8** (`src-tauri/src/db.rs`), each one focused, matching the v1–v4 style:
- **v5 `create_projects`** — `projects` keyed on `canonical_key` (git remote `owner/repo`, folder-name fallback) **UNIQUE** + `project_aliases(alias_kind ∈ folder|title|github-url, alias_value)` with `UNIQUE(alias_kind, alias_value)`. The unique constraints are what enforce D30's *one project, never fragmented*.
- **v6 `create_sites`** — `sites(host UNIQUE, display_name, kind ∈ general|dashboard|project-host|unknown)` registry; supports Phase-2 correlation of `dashboard`/ambiguous browsing.
- **v7 `create_exclusions`** — `exclusions(match_type ∈ app|site|title, pattern, mode ∈ exclude|private)` (D8).
- **v8 `add_event_enrichment_columns`** — `activity_logs += url, site, project_id (FK), project_abstain_reason (NULL|no-signal|ambiguous), is_private` + index.

**Typed repository** (hard rule 4): `resolve_or_create_project`, `find_project_by_alias`, `add_project_alias`, `get/delete_project`, `resolve_or_create_site` / `set_site_kind` / `get_sites`, exclusion CRUD + `match_exclusion` (returns the strongest match — `Exclude` > `Private`), and `insert_event`/`NewEvent` (the capture/enrichment write path). The legacy `insert_activity_log`/`log_activity` path is untouched, so the current watcher keeps working and writes empty enrichment.

**Other:** WAL enabled in `init_database` (R57; writer thread still deferred). `db` made a `pub mod` so the not-yet-wired repository API isn't `dead_code` under `-D warnings`.
